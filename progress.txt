## Session (2026-01-31 afternoon) - Wagmi Mock Connector Research (COMPLETED)

**Task:** Research wagmi connector capabilities for private key account support in E2E tests

**Key Findings:**

1. **Wagmi mock connector ONLY accepts addresses, NOT private keys**
   - Parameter type: `accounts: readonly [Address, ...Address[]]`
   - Documentation confirms it takes address strings (0x...), not private key accounts
   - Source: https://wagmi.sh/core/api/connectors/mock

2. **Wagmi does NOT support hoisting private key accounts to wagmiConfig**
   - This is a known limitation discussed in GitHub issue #3480
   - Quote: "Wagmi currently does not support hoisting Private Key & Mnemonic Accounts to the top-level Wagmi Config"
   - Workaround mentioned: "creating a custom connector based on Viem account"
   - Source: https://github.com/wevm/wagmi/discussions/3480

3. **Current implementation already converts private keys correctly**
   - `ui/src/wagmi.ts` createMockConfig() function (lines 33-60):
     - Accepts `addressOrPkey: 0x${string}`
     - Converts private key to account using `privateKeyToAccount(addressOrPkey)`
     - But then ONLY passes the address to mock connector: `mock({ accounts: [address] })`
     - The actual account object with signing capability is discarded!

4. **The root problem:**
   - SDK actions expect `walletClient.account` to be a full viem Account object with signing methods
   - Wagmi's useWalletClient() hook returns a wallet client from the connected connector
   - Mock connector only knows about the address, not the private key
   - Therefore walletClient.account is either undefined or a minimal object without signing capability

5. **Why approach #3 (from previous session) won't work:**
   - The previous session recommended "Update wallet fixture to use private key accounts"
   - But wagmi's mock connector fundamentally doesn't support this
   - Even if we pass the full account object to createMockConfig, we can only pass the address to mock()
   - The wallet client returned by useWalletClient() will still not have signing capability

**Conclusion:**

Approach #3 from the previous session is **NOT viable**. The wagmi mock connector cannot provide a wallet client with private key signing capabilities.

**Recommended new approach:**

We need to create a **custom connector** that wraps a viem account with private key:

1. Create `ui/e2e/fixtures/privateKeyConnector.ts`:
   - Custom wagmi connector that holds a full viem account (from privateKeyToAccount)
   - Implements the Connector interface
   - Provides getProvider(), getAccounts(), etc.
   - Most importantly: makes the full account object available to walletClient

2. Use the custom connector in createMockConfig instead of mock():
   ```typescript
   connectors: [createPrivateKeyConnector(privateKey)]
   ```

3. This matches the workaround mentioned in GitHub discussions: "creating a custom connector based on Viem account"

**Alternative approach (potentially simpler):**

Instead of fighting with wagmi, modify the UI components to NOT use the SDK actions directly. Instead:
- Use wagmi's native hooks: `useWriteContract()`, `useSimulateContract()`, etc.
- These hooks handle signing through the connector properly
- Create a thin adapter layer that translates SDK action patterns to wagmi patterns
- Keep SDK actions as-is for integration tests (which use createTestClients with real accounts)

This second approach is more aligned with wagmi's architecture and may be more maintainable.
