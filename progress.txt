# Integration Tests - TypeScript Type Error Fix (statementId → statementCid refactor)

## Task
Run `npm run build --workspace=integration-tests` and fix type errors. The codebase is
mid-refactor: `statementId` (string/bytes32) is being renamed to `statementCid` (IpfsCidV1).

Key types:
- `IpfsCidV1 = \`b${string}\`` — IPFS CIDv1, starts with "b"
- `IpfsCidBytes32 = \`0x${string}\`` — bytes32 hex, for on-chain storage
- `cidToBytes32(cid: IpfsCidV1): IpfsCidBytes32` — converts CID → bytes32 (for on-chain use)
- GraphQL/indexer helpers now expect `IpfsCidV1`, not bytes32

## Core Pattern Being Fixed
Old code created both:
  `const someCid = await uploadToIPFS(...)   // IpfsCidV1`
  `const someId = cidToBytes32(someCid)      // IpfsCidBytes32`
...then passed `someId` to graphql helpers. New code: pass `someCid` directly.

## KNOWN ISSUE (needs design decision, don't try to "fix" deeply)

`PROJECT_ALIGNMENT_TOPIC` is exported from the SDK as `0x${string}` (a keccak256 hash),
but `attestAlignmentChecked` expects `topicStatementCid: IpfsCidV1` as its 6th argument.
These are fundamentally different things — a keccak hash is NOT an IPFS CID.

The workaround applied throughout is: `PROJECT_ALIGNMENT_TOPIC`

This suppresses the TypeScript error but the runtime behavior should be reviewed. Either:
a) `PROJECT_ALIGNMENT_TOPIC` should be redefined as a proper `IpfsCidV1` in the SDK, or
b) `attestAlignmentChecked` / the alignment contract should accept a separate `topicId` type
