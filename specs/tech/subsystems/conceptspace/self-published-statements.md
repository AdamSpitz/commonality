# Self-published statements (calldata in the signing transaction)

Status: **proposed / not yet scheduled** (Jul 2026). This is a design direction, not a commitment. The legal motivation lives in [specs/product/legal/statement-hosting.md](/specs/product/legal/statement-hosting.md); this file is the technical side. [eliminating-ipfs.md](/specs/tech/eliminating-ipfs.md) generalizes this design to every other IPFS use in the project.

## The idea

Today, statement content lives on IPFS and *we* pin it. The proposal: the statement's bytes ride in the **calldata of the author's own transaction**, signed with their key and paid with their funds. No intermediary occupies the publish path: no pinning node we run, no upload service, no storage account we fund.

This is the well-proven "inscription" pattern (Ethscriptions, Ordinals). Statements are a few KB of markdown; on a post-4844 L2 the marginal cost of a few KB of calldata is fractions of a cent (dominated by blob-priced L1 data availability) — "a few cents" is a safe ceiling even in fee spikes. IPFS was never buying meaningful cost savings; it was buying a hosting role we don't want.

## Why the transaction, and not a storage network

The legal analysis (see [statement-hosting.md](/specs/product/legal/statement-hosting.md)) wants the *author* — not us, not the vertical — to be the publisher and host of each statement. The storage options divide on one axis that turns out to be decisive: **pay-once vs. rent.**

- **Rent-based storage recreates the role we're trying to vacate.** IPFS pinning, Swarm postage stamps, and Filecoin deals all require an *ongoing payer*. Someone must keep paying for each statement, and "the operator tops it up for convenience" puts us right back in the hosting seat. Swarm is alive but niche (active Bee development, thin adoption); Filecoin deals expire and need renewal. Neither changes the legal shape.
- **Pay-once storage lets the user's single act complete everything.** Arweave has the right semantics but is outside the Ethereum ecosystem and — critically — if *we* did the uploading, permanence at a layer we operate is actively bad (see the "engineered incapacity" analysis in statement-hosting.md). Calldata has pay-once semantics natively, inside the same transaction that signals belief.
- **Blobs (EIP-4844) directly: no.** They expire after ~18 days by design.
- **EthStorage: the watch-list upgrade path.** A storage L2 where data is posted via blob transactions and then permanently replicated by incentivized storage providers (~0.1% of L1 cost; Optimism grant to serve as long-term DA backstop for OP Stack chains). It reached Mainnet Alpha in 2026 — the closest Ethereum-native Arweave analog, but too young to make load-bearing yet. Natural future home if statements outgrow calldata.

## Design sketch

- **Identity stays CID-shaped.** The client computes the IPFS-style CID of the content locally, the contract event carries the CID (as today), and the bytes are recoverable from the transaction by anyone. Nothing downstream of the event schema needs to change its notion of statement identity.
- **A separate publish operation carries the bytes; `supportStatement` references the CID only.** The first person who wants a statement to exist pays once (the few cents) to publish its content; every subsequent support is an ordinary cheap transaction with ~36 bytes of args. This beats the alternative ("the first `DirectSupport` carries the content") on several axes at essentially identical cost:
  - **No race/duplication ambiguity.** With content-in-first-support, two near-simultaneous first supporters either both pay to include the bytes or the client has to check-and-hope. With an explicit publish operation, the contract stores a publication bit per (publisher, cid) (~20k gas) and rejects or no-ops duplicates from the same publisher.
  - **Onchain verification becomes practical.** Since the contract sees the bytes, it can verify content-matches-CID (keccak/sha256 precompiles are cheap at a few KB), so the publication bit actually means something — and `supportStatement` can require the statement exists, giving free typo/garbage-CID protection.
  - **Statements can exist unsigned.** Negations, bridge-creator statements, a vertical's seed cluster, and statements created only to be referenced by an implication all need to exist without any believer. Content-in-first-support has no home for those.
  - **The common human flow is unaffected.** The composer UI can still make "publish and sign" one click — two transactions or a create+support multicall — so the "single act authors, publishes, and attests" story survives for the typical author.
  - **Support does not require prior publication (resolved Jul 2026).** `supportStatement` is a pure attestation about a CID and never gates on `isPublished`; a support whose CID has no honored live publication renders as "supports an unavailable statement" (threaded-forum semantics — a reply survives the parent's deletion). *Advanced option, noted but deliberately not built yet:* let a supporter optionally (re)publish the bytes as part of supporting — becoming an additional `(publisher, cid)` — so their support also keeps the statement alive if the original publisher later retracts. Too complex for the normie flow; keep it out of the main composer UI.
- **Creating and believing are distinct acts.** The creator of a statement is its *publisher*, not necessarily a *believer*; belief is attested only by `DirectSupport`. Legally this is fine — attribution still lands on the address that chose to publish. [statement-discovery.md](statement-discovery.md) discovers statements via `DirectSupport` events; discovery can stay support-driven (an unsigned statement is discoverable via whatever references it) or additionally index PublishedData's `DataPublished` events — an open detail for scheduling time.
- **IPFS is demoted from canonical storage to optional retrieval cache.** Verticals pin the statements they curate (their core statement S, their seed cluster, their explorer's curated-collection entries); authors and communities pin whatever they care about; published pin-lists invite re-pinning. Nobody's pin is load-bearing for availability, and each pinner's pins match their editorial exposure.
- **Indexers cache content from history.** Each operator's indexer (already intended to be operator-scoped — see [ui-operator-posture.md](/specs/product/ui-operator-posture.md)) fetches statement bytes from chain history for the statements it recognizes, and serves them to its UI. Re-serving is that operator's curated display act, subject to its denylist.
- **Publication and retraction are generalized into the [PublishedData subsystem](../published-data/README.md).** Rather than baking publish/verify/retract into the Beliefs contract, statements route through the shared `PublishedData` contract: `publishData(bytes content)` verifies content-matches-CID and records the publication for **(publisher, cid)** — the primitive is "Alice published this," not "this exists" — and the onchain `isPublished` view exists for any consumer that wants it, but `supportStatement` deliberately does *not* gate on it (resolved Jul 2026 — support is a pure attestation about a CID). The composer still offers publish+support as one click for the common author flow.
- **Retraction, not erasure.** A publisher can record an onchain retraction of their own publication — a permanent "I take this back" mark, not a deletion. Conforming display layers and aggregators honor it by default: hide the statement, stop counting its supports in aggregates, show "retracted by author" where a reference would otherwise render it. This is the author's compliance lever for the permanence problem (see the legal caveats below and in [statement-hosting.md](/specs/product/legal/statement-hosting.md#permanence-cuts-back-the-users-side-of-the-bargain)): faced with a removal order or their own change of heart, the author can do everything within their power — retract onchain, knowing every conforming display suppresses it. Retraction is keyed to the publisher (Alice retracting her publication of X doesn't burn X for other publishers), distinct from support withdrawal (supporters just withdraw support), and distinct from any operator's denylist — though the denylist itself generalizes into onchain retraction attestations honored per display-layer policy; see [PublishedData](../published-data/README.md#honored-retractors-the-denylist-generalized). Nudger revocations already establish the "publish a signed take-back, displays honor it" pattern.

## Honest caveats

- **Chain history is socially, not cryptographically, permanent — and getting more prunable.** Partial history expiry (EIP-4444) went live for pre-merge data in 2025, with rolling-window expiry planned in a 2026 hardfork; long-term retrieval shifts to archive nodes and the Portal Network. On an L2 it's a notch weaker still: the L1 blobs the L2 posts expire after ~18 days, so durability rests on L2 nodes retaining their own history. In practice, many independent parties (including every vertical's indexer and pinner) retain the statements anyone cares about. Note that the *legal* story doesn't depend on permanence at all — it depends on who published.
- **Permanence creates real legal exposure for the *user* — the "ordered to take down but can't" trap.** A user who loses a defamation suit and is ordered to remove their statement literally cannot comply. Impossibility generally defends against contempt, but the impossibility was the user's own prior choice, and irremediable ongoing publication can inflate damages. Worse is the third-party case: a statement that doxxes or defames *someone else* embeds the victim's personal data permanently — the most sympathetic plaintiff facing the most trapped defendant. And in regimes with blasphemy/lèse-majesté/anti-dissent laws, a statement is permanent self-incriminating evidence even if the user recants or flees. The retraction mechanism (above) is the answer at the only layer that matters: in practice "take it down" means "stop people from seeing it," and nearly all real harm flows through rendering, aggregation, and discovery — layers that *can* comply. Retraction converts "literally can't comply" into "complied at every layer that renders"; the residue is closer to "an archived court record exists" than "still published."
- **GDPR is the one place the incapacity argument can be turned against us.** Statements of belief are Article 9 special-category data (political/religious opinions) tied to a persistent identifier; once the address links to an identity, the user has published sensitive personal data no erasure right can reach — and GDPR consent is *withdrawable*, so informed consent alone doesn't close this. The EDPB's 2025 draft blockchain guidelines lean toward "don't put personal data on-chain" and will find controllership in whoever *designs the processing* — and we build the composer, the contract, and the flow. Mitigations: addresses stay pseudonymous (we never require identity linkage), the Art 17(3)(a) freedom-of-expression exception (it's the user's own expressive speech), we process nothing ourselves, and retraction honors withdrawal-of-consent at the display layer. But note the tension: the same "we designed it so nobody can comply" that protects us in the hosted-speech frame can hurt us in the privacy frame.
- **Informed consent is required UX — and must be blunter than "cannot be deleted."** The composer flow must say, in substance: *you may someday be legally ordered to remove this, and you will not be able to — retraction (which hides it from conforming displays but leaves it in chain history) is all you will ever have.* That converts the privacy downside of permanence from our design decision into the user's genuinely disclosed choice. The composer is also the right place for a third-party-personal-data friction step: not a filter we enforce, but a "does this contain personal information about an identifiable person?" warning the user acknowledges — it protects the doxxing victim's interests at the cheapest point and strengthens the consent record.
- **Vile content in calldata is survivable.** Someone will eventually push something awful through `createStatement` (or raw, bypassing our UI). Chain history containing illegal bytes is an old story (Bitcoin's OP_RETURN has carried illegal content for a decade; no chain has been forced to purge), and the text-only restricted format limits the worst categories. Our real exposure is the indexer/display layer caching and re-serving it — which the denylist covers, and which is precisely the lever we keep on purpose.
- **Gas sponsorship would undo the point.** If an operator sponsors the publish transaction, it's back on the facilitation surface ([political-funding.md](/specs/product/legal/political-funding.md) in-kind angle). Users paying their own gas to publish their own beliefs is both the cleanest legal fact and decent spam resistance. In particular, the UI must never offer "we'll `createStatement` for you on our dime" — operator-paid creation of user-drafted statements is exactly the we-publish-what-you-tell-us-to-post role, in miniature.
- **Machine-created statements: we really are the publisher, and that's the correct outcome.** For statements our agents create (bridge statements, explorer negations, a vertical's core S and seed cluster), the paying/signing key is ours or the vertical's — for those, we genuinely are author and publisher. That's consistent with the existing posture (the vertical's S is already its own editorial speech, owned the way Civility owns its criteria). `createStatement` doesn't vacate the publisher role universally; it makes the role land on whoever actually authored, which is the point — and it keeps the "LLM drafts → a human/vertical explicitly signs and owns it" principle load-bearing.
- **The composer tool is a residual role.** Someone still runs the UI where users write statements. Providing a tool is far better than publishing — the user signs and pays — but an LLM that *drafts* statement text has a co-authorship flavor. The existing principle from [explorer.md](explorer.md) ("the LLM never generates statement text directly — it references existing statements by CID, or creates a new statement and then references it" — and statement creation surfaces the text for the *user* to sign) is doing quiet legal work; keep it.

## What would need to change

1. Build the [PublishedData subsystem](../published-data/README.md): the shared contract (`publishData`, `retractData`, existence/retraction views) and the SDK reader library with its status-discriminated return type.
2. The statement-creation flow: build content into a `publishData` transaction's calldata instead of (or alongside) uploading to IPFS; compute the CID client-side; offer publish+support as one click for the common author flow. `supportStatement` does **not** require `isPublished` (resolved Jul 2026 — support is a pure attestation about a CID, threaded-forum semantics; existence stays a display-layer concern). See the [PublishedData readiness note](../published-data/README.md#readiness-note-design-resolved-except-two-technical-benchmarks-jul-2026).
3. Indexers: ingest `DataPublished`/`DataRetracted`, extract statement bytes from publication calldata (falling back to IPFS gateways for legacy statements), and expose retraction status so UIs and aggregators can suppress retracted statements by default.
4. Pinning posture: publish pin-lists; verticals pin what they curate; drop the assumption that "we pin everything."
5. UIs/aggregators: honor retractions by default (hide, stop counting, show "retracted by author" at reference sites); composer consent language per the caveats above, including the third-party-personal-data warning.
