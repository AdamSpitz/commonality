What the app seems to be for: it’s a coordination and funding platform for people with shared values. Users express beliefs as statements, the system connects related beliefs, and then helps route money toward aligned projects or creators. The docs page made this much clearer than the home page. In plain English: “find people who care about the same thing, discover projects/content that fit that cause, and fund them directly, via pledges, or via trusted delegates.”

Does it make sense? Mostly yes, once I reached `#/docs`. That page is strong and gave me a coherent model:
- statements are the entry point
- the implication graph connects related causes
- assurance contracts mean “fund only if threshold is met”
- delegation means “let someone I trust allocate funds”
- content funding works for creators on Twitter/YouTube/Substack
- trust networks filter whose attestations you rely on

Where it’s still confusing for a new user:
- The home page is too vague. “track their numbers” doesn’t help much.
- The nav throws a lot of jargon at you immediately: `My Notes`, `Refs`, `Trusted Implication Attesters`, `Alignment Trust Network`.
- `Settings` is very advanced and reads like internal machinery, not onboarding.
- Several data-backed pages looked empty or errored with `Failed to fetch events: 404 Not Found`, so the product can feel broken before a user understands it.
- The most helpful explanation is in `Docs`, but that doesn’t feel like the primary first-run path.

If I were explaining it to a new user, I’d say:

“Commonality helps people fund projects and content around shared values. You start by expressing what you care about through statements. The system then connects you to related causes, projects, and creators. You can pledge directly, delegate to someone you trust, or support creators whose work fits those causes. The key idea is that coordination and trust are built into the system, so people can support things together without needing a central organization.”

Best “how to get started” flow for a new user:
1. Open `Docs` first.
2. Read one walkthrough, especially the political-writing or block-party example.
3. Go to `Browse Statements` to see the causes/beliefs layer.
4. Connect wallet only after the user understands why they’d want to.
5. Then explore creator/funding pages.

My honest take: the product idea does make sense, and the docs communicate it pretty well, but the app’s default UX still feels like “power user/product internals first” rather than “new user first.” The main opportunity is to turn the docs explanation into the actual onboarding path.

## AI's response

Yes. The main friction wasn’t understanding the product, it was just getting to the right rendered app reliably.

What tripped me up:
- `./services.sh --start` did work, but it didn’t print the final URL back into my tool session before the command detached/finished. I had to read `data/ui-ipfs/gateway-url.txt`.
- The URL in that file points at the IPFS root directory, not the SPA subdirectory. The root showed the Kubo directory listing first, and I had to click into `/commonality-ui/`.
- From inside my sandbox, `localhost:8080` wasn’t reachable even though Docker had published the port. I had to use escalated commands to fetch it.
- Docker status checks also needed escalation because the sandbox couldn’t talk to `/var/run/docker.sock`.

If you want this to be smoother for a fresh LLM, I’d recommend:
- Make `services.sh --start` always print one final, explicit line like: `SPA URL: http://localhost:8080/ipfs/<cid>/commonality-ui/#/`
- Write that exact SPA URL, not the parent IPFS directory URL, into `data/ui-ipfs/gateway-url.txt`
- Optionally also write a tiny helper file like `data/ui-ipfs/open-url.txt` or `data/ui-ipfs/spa-url.txt` so there’s zero ambiguity
- Add a quick `./services.sh --url` mode that only prints the live SPA URL
- Add a short readiness check so `--start` doesn’t return until the gateway serves the SPA entrypoint successfully
- In `README.md`, say explicitly that the published bundle lives under `/commonality-ui/#/`, not just under the raw CID root

If you’re specifically optimizing for “fresh LLM pointed at local app,” the single biggest win is: give it one canonical URL that is already the real SPA route and not an IPFS directory listing.
